# How-to: Generate Java Bindings with jextract

**Diátaxis quadrant: How-to** — Task-oriented. You have a C header; you need a Java FFI class.

---

## Goal

Run `jextract` against your C façade header to produce a Java class with
`FunctionDescriptor` constants and `MethodHandle` factories, then know what to
do when jextract is unavailable.

---

## Prerequisites

- jextract 21+ installed. Verify: `jextract --version`
- Your C façade header is complete (see [How-to: Write a C Façade](write-c-facade.md))
- The header uses only C types (no C++ templates, no `std::` types)

---

## Step 1 — Run jextract

```bash
jextract \
  --output src/main/java \
  --target-package org.yawlfoundation.yawl.mylib.ffi \
  --library my_lib_ffi \
  src/main/native/my_lib_ffi.h
```

| Flag | Meaning |
|------|---------|
| `--output` | Root of your Java source tree |
| `--target-package` | Package for the generated class |
| `--library` | Name passed to `System.loadLibrary()` |
| Last arg | Path to your C header |

This produces:
```
src/main/java/org/yawlfoundation/yawl/mylib/ffi/
├── my_lib_ffi_h.java           ← main bindings class
├── MyStatus.java               ← MemoryLayout for the struct
└── constants/                  ← #define constants as Java fields
```

---

## Step 2 — Review the Generated Class

Open `my_lib_ffi_h.java` and verify:

1. **One `FunctionDescriptor` field per function** with the correct type
   sequence.
2. **One `MethodHandle` factory method per function** (e.g.,
   `my_lib_ffi_h.my_thing_create()`).
3. **No C++ names** in the output. If you see mangled names the header is not
   pure C.

Example expected output for `MyThing* my_thing_create(int64_t v, MyStatus* s)`:

```java
public static final FunctionDescriptor my_thing_create$FUNC =
    FunctionDescriptor.of(
        ValueLayout.ADDRESS,      // MyThing* (return)
        ValueLayout.JAVA_LONG,    // int64_t v
        ValueLayout.ADDRESS       // MyStatus* s
    );

public static MethodHandle my_thing_create() {
    return SymbolLookup.loaderLookup()
        .find("my_thing_create")
        .map(addr -> Linker.nativeLinker().downcallHandle(
            addr, my_thing_create$FUNC))
        .orElseThrow(() ->
            new UnsatisfiedLinkError("my_thing_create"));
}
```

---

## Step 3 — Keep the Generated File

Commit `my_lib_ffi_h.java` to version control. Do **not** regenerate it on
every build — jextract output is stable for a given header, and regenerating
during CI introduces non-determinism.

Add a comment at the top:

```java
// Generated by jextract from src/main/native/my_lib_ffi.h
// DO NOT EDIT — regenerate by running: scripts/run-jextract.sh
```

Write `scripts/run-jextract.sh` documenting the exact command.

---

## Writing Bindings by Hand (When jextract Is Unavailable)

jextract is a convenience tool; it produces nothing you cannot write manually.
Use this template for each C function:

```java
// C: ReturnType fn_name(ArgType1 a, ArgType2 b, ...);
public static final FunctionDescriptor fn_name$FUNC =
    FunctionDescriptor.of(         // use FunctionDescriptor.ofVoid() for void
        <return ValueLayout>,
        <arg1 ValueLayout>,
        <arg2 ValueLayout>
    );

private static volatile MethodHandle fn_name$MH;   // lazy + volatile

public static MethodHandle fn_name() {
    if (fn_name$MH == null) {
        fn_name$MH = SymbolLookup.loaderLookup()
            .find("fn_name")
            .map(addr -> Linker.nativeLinker().downcallHandle(
                addr, fn_name$FUNC))
            .orElseThrow(() ->
                new UnsatisfiedLinkError("fn_name not found in libmy_lib_ffi"));
    }
    return fn_name$MH;
}
```

See [Reference: Type Mapping](../reference/type-mapping.md) for the full
C-type → `ValueLayout` translation table.

---

## Handling `void` Return Functions

`FunctionDescriptor.ofVoid()` for functions with no return value:

```java
// C: void my_thing_destroy(MyThing* t);
public static final FunctionDescriptor my_thing_destroy$FUNC =
    FunctionDescriptor.ofVoid(ValueLayout.ADDRESS);
```

---

## Caching MethodHandles

Resolve all `MethodHandle` objects **once** in the constructor of your bindings
class. Never call `counter_ffi_h.fn_name()` inside a hot path:

```java
public final class MyFfiBindings implements AutoCloseable {
    private final MethodHandle mhCreate;
    private final MethodHandle mhDestroy;
    // ...

    public MyFfiBindings() {
        this.mhCreate  = my_lib_ffi_h.my_thing_create();   // once
        this.mhDestroy = my_lib_ffi_h.my_thing_destroy();  // once
    }
}
```

`MethodHandle` resolution involves a native symbol lookup. Caching eliminates
that overhead from every call.

---

## Linker Options for Performance

For functions on the critical path, pass `Linker.Option.critical(false)`:

```java
this.mhQuery = Linker.nativeLinker().downcallHandle(
    SymbolLookup.loaderLookup().find("my_thing_exec").orElseThrow(),
    my_lib_ffi_h.my_thing_exec$FUNC,
    Linker.Option.critical(false)   // avoids thread-state save/restore
);
```

`critical(false)` tells the linker the native function does not call back into
Java. This reduces overhead on calls that are provably safe. Do not use it for
functions that take callbacks or that call JNI/FFM callback handles.

---

## Checklist

- [ ] Every `extern "C"` function in the header has a matching `$FUNC` and factory method
- [ ] `void` returns use `FunctionDescriptor.ofVoid()`
- [ ] Non-void returns have the return layout as the first argument to `FunctionDescriptor.of()`
- [ ] Generated file is committed, not regenerated every build
- [ ] MethodHandles are cached in the bindings constructor
- [ ] `UnsatisfiedLinkError` message names the missing symbol and the library
