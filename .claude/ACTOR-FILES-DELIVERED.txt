## Design Decisions Documented

1. ActorRef as value type (not exposing Agent)
   → See: ACTOR-INTERFACES.md → Design Principles
   → Rationale: Immutability, opaque handle, value semantics

2. Sealed Msg hierarchy (not open classes)
   → See: ACTOR-INTERFACES.md → Message Hierarchy Design
   → Rationale: Zero-overhead polymorphism, exhaustive matching

3. One VirtualThread per actor (not thread pool)
   → See: ACTOR-INTERFACES.md → Supervisor Architecture
   → Rationale: Simple, scales to millions, optimal for blocking

4. Supervisor wraps Runtime (not separate orchestrator)
   → See: ACTOR-INTERFACES.md → Supervisor Architecture
   → Rationale: Local restart policy, cleaner API, lifecycle ownership

5. 4-byte int ID (not 16-byte UUID)
   → See: ACTOR-INTERFACES.md → Byte Accounting
   → Rationale: Saves 12 bytes per ActorRef, sufficient range (4B)

6. LinkedTransferQueue mailbox (not bounded array)
   → See: ACTOR-INTERFACES.md → Hot Path Optimization
   → Rationale: Lock-free, optimal for blocking, no GC resizing

## Performance Targets

Tell throughput:
  Target: > 1M msgs/sec
  Expected: ~10M msgs/sec (lock-free path)

Ask latency:
  Target: < 10 µs (median)
  Expected: ~1-5 µs (application-dependent)

Spawn throughput:
  Target: > 100K actors/sec
  Expected: ~1M actors/sec (virtual thread creation is cheap)

Memory per idle actor:
  Target: < 200 bytes
  Actual: ~152 bytes (with -XX:+UseCompactObjectHeaders)

At 10M scale:
  Memory usage: ~1.5 GB (vs Erlang: 18.6 GB)
  Throughput: >100M messages/sec possible
